# Техническое задание: Backend (Low-Code Platform)

## 1. Общие сведения
**Наименование проекта:** Универсальный Low-Code Backend на Python.
**Цель:** Разработка серверной части платформы, позволяющей пользователям создавать модели данных, настраивать бизнес-логику и генерировать API через веб-интерфейс без написания кода.
**Архитектурный стиль:** Модульный монолит с элементами микросервисной архитектуры (разделение API, Workers, Core), упакованный в Docker Compose.

## 2. Стек технологий (Универсальный вариант)
*   **Язык:** Python 3.10+
*   **Web Framework:** FastAPI (или аналогичный асинхронный фреймворк) для высокого параллелизма.
*   **ORM / Data Layer:** SQLAlchemy 2.0+ (Async) + Alembic для миграций. Реализация паттерна "Dynamic Model Registry".
*   **СУБД:** PostgreSQL (основное хранилище конфигураций и пользовательских данных).
*   **Cache / Broker:** Redis (кеширование метаданных, очередь задач для воркеров).
*   **Task Queue:** Celery или ARQ (для выполнения тяжелой бизнес-логики и триггеров).
*   **Контейнеризация:** Docker & Docker Compose.

## 3. Архитектура системы
Система состоит из следующих логических модулей, развертываемых как отдельные процессы/контейнеры:

1.  **API Gateway Service:** Принимает входящие запросы, выполняет аутентификацию, роутинг к динамическим контроллерам.
2.  **Core Engine:** Ядро системы. Управляет метаданными, схемой БД, валидацией.
3.  **Worker Service:** Фоновые воркеры для выполнения триггеров, хуков и сложной бизнес-логики.
4.  **Migration Service:** Утилита для применения изменений схемы БД при обновлении моделей пользователя.

## 4. Функциональные требования по модулям

### 4.1. Модуль управления метаданными (Metadata Engine)
*   **Хранение схем:** Все определения моделей (поля, типы, связи, индексы) хранятся в специальной системной таблице `system_models` в формате JSONB.
*   **Динамическая генерация моделей:** При старте приложения или по сигналу обновления система должна читать JSONB и динамически создавать SQLAlchemy-модели в памяти.
*   **Миграции:** Автоматическая генерация и применение миграций (ALTER TABLE, CREATE TABLE) при изменении структуры модели пользователем.
*   **Валидация схем:** Проверка корректности имен полей, типов данных и циклических зависимостей перед сохранением.

### 4.2. Динамический API слой
*   **CRUD Endpoints:** Автоматическая генерация стандартных маршрутов для каждой созданной пользователем модели:
    *   `POST /api/v1/{model_name}` — создание
    *   `GET /api/v1/{model_name}` — список (с фильтрацией, сортировкой, пагинацией)
    *   `GET /api/v1/{model_name}/{id}` — получение детали
    *   `PUT/PATCH /api/v1/{model_name}/{id}` — обновление
    *   `DELETE /api/v1/{model_name}/{id}` — удаление
*   **Фильтрация и поиск:** Поддержка сложных запросов через query-параметры (например, `?filter[status]=active&sort=-created_at`).
*   **Кастомные эндпоинты:** Возможность привязки пользовательских скриптов (Python-код или визуальный блок) к конкретным HTTP-методам и путям.

### 4.3. Движок бизнес-логики (Logic Engine)
*   **Триггеры:** Система хуков `Before Create`, `After Create`, `Before Update`, `After Delete`.
*   **Исполнение кода:** Безопасное выполнение пользовательской логики (в MVP — доверенный Python код в песочнице, в будущем — WebAssembly или ограниченный DSL).
*   **Очереди задач:** Асинхронное выполнение длительных операций через Redis/Celery.
*   **Планировщик:** Встроенный Cron-планировщик для периодических задач.

### 4.4. Система безопасности и доступа (Auth & RBAC)
*   **Аутентификация:** JWT (Access + Refresh tokens). Поддержка OAuth2 провайдеров (опционально).
*   **Ролевая модель (RBAC):**
    *   Системные роли: `Admin`, `Developer`, `Viewer`.
    *   Пользовательские роли: Создаются администратором.
*   **Разрешения (Permissions):** Гранулярная настройка прав (Create, Read, Update, Delete) на уровне *Модель -> Роль*.
*   **Полевая безопасность:** Возможность скрывать определенные поля модели от конкретных ролей.

### 4.5. Аудит и логирование
*   **Лог действий:** Запись всех изменений данных (кто, когда, что изменил, старые и новые значения) в таблицу `audit_logs`.
*   **Логи ошибок:** Детальное логирование ошибок выполнения скриптов и запросов.

## 5. Нефункциональные требования
*   **Производительность:** Время отклика API < 200мс для простых CRUD операций (без учета сложной логики).
*   **Масштабируемость:** Возможность горизонтального масштабирования API и Worker узлов через Docker Compose / K8s.
*   **Изоляция:** Падение пользовательского скрипта не должно валить основной процесс API.
*   **Типизация:** Строгая типизация кода (Type Hints), использование Pydantic для валидации входных/выходных данных.

## 6. Структура репозитория (Backend)
```text
/backend
├── app
│   ├── core          # Конфигурация, безопасность, зависимости
│   ├── models        # Системные модели (User, Role, ModelDefinition)
│   ├── dynamic       # Логика динамической генерации моделей и ORM
│   ├── api           # Роутеры (System API, Dynamic API)
│   ├── services      # Бизнес-логика ядра
│   ├── workers       # Задачи Celery/ARQ
│   └── utils         # Утилиты
├── tests             # Юнит и интеграционные тесты
├── Dockerfile
├── requirements.txt
└── alembic           # Миграции БД
```

## 7. Этапы разработки (Roadmap)
1.  **MVP (Спринт 1):**
    *   Настройка Docker Compose (Postgres, Redis, API).
    *   Реализация системных моделей (пользователи, роли).
    *   Движок чтения метаданных и генерации простых CRUD.
2.  **Спринт 2:**
    *   Реализация динамических миграций БД.
    *   Внедрение фильтров, сортировок и пагинации.
    *   Базовая система прав доступа (RBAC).
3.  **Спринт 3:**
    *   Интеграция очереди задач (Celery).
    *   Реализация триггеров (Pre/Post hooks).
    *   Система аудита.
4.  **Спринт 4:**
    *   Оптимизация производительности, кеширование схем.
    *   Документирование API (OpenAPI/Swagger).
    *   Нагрузочное тестирование.

## 8. Критерии приемки
*   Система разворачивается одной командой `docker-compose up -d`.
*   Через API можно создать новую модель данных, и она сразу доступна для запросов.
*   Изменение модели (добавление поля) автоматически применяет миграцию БД.
*   Права доступа корректно ограничивают доступ к данным.
*   Покрытие тестами критического функционала > 80%.
