# Техническое задание: Backend (Low-Code Platform Core)

## 1. Обзор проекта
**Цель:** Создание универсального low-code backend на Python, предоставляющего API для динамического управления структурой данных (сущности, поля) и автоматической генерации CRUD-операций, а также выполнения бизнес-логики через workflow.

**Ключевая архитектурная особенность:** `JSONB-first` подход. Вместо динамической генерации таблиц и классов ORM используется единая схема хранения записей с гибкой структурой в поле JSONB PostgreSQL. Это исключает необходимость миграций схемы при изменении модели данных пользователем.

## 2. Технологический стек
| Компонент | Технология | Обоснование |
| :--- | :--- | :--- |
| **Язык** | Python 3.11+ | Производительность async, улучшенная типизация |
| **Web Framework** | FastAPI | Async из коробки, авто-генерация OpenAPI, скорость |
| **DB Driver** | SQLAlchemy Core + asyncpg | Прямой контроль SQL, эффективная работа с JSONB без оверхеда ORM |
| **Валидация** | Pydantic v2 | Валидация входящих данных и метаданных схем |
| **СУБД** | PostgreSQL 15+ | Нативная поддержка JSONB, GIN-индексы, транзакции |
| **Кэш / Брокер** | Redis | Кэширование метаданных, очередь задач |
| **Task Queue** | ARQ | Легковесная async-очередь на базе Redis (проще Celery) |
| **Auth** | python-jose / Authlib | JWT токены, управление сессиями |
| **Логирование** | Structlog | Структурированные JSON-логи для наблюдаемости |
| **Контейнеризация** | Docker & Docker Compose | Единая точка запуска всей инфраструктуры |

## 3. Архитектура данных (Data Model)

### 3.1. Системные таблицы (Metadata)
Хранят конфигурацию приложения, созданную пользователем через UI.

1.  **`entities`** (Определение сущностей)
    *   `id` (UUID): Уникальный ID сущности.
    *   `slug` (String): URL-safe имя (например, `orders`, `users`).
    *   `name` (String): Отображаемое имя.
    *   `schema_definition` (JSONB): Описание полей (имя, тип, валидация, обязательность).
        *   Пример: `{"fields": [{"key": "price", "type": "number", "required": true}, ...]}`
    *   `created_at`, `updated_at`: Таймстампы.
    *   `version` (Integer): Версия схемы для контроля консистентности.

2.  **`roles` & `permissions`** (RBAC)
    *   Роли привязываются к сущностям и действиям (`create`, `read`, `update`, `delete`).
    *   Поддержка фильтров уровня строк (Row-Level Security): `{"filter": {"owner_id": "$user.id"}}`.

### 3.2. Таблица данных (Data Storage)
Единая таблица для хранения всех записей всех сущностей.

1.  **`records`**
    *   `id` (UUID): Уникальный ID записи.
    *   `entity_id` (UUID): Ссылка на `entities.id`.
    *   `tenant_id` (UUID): Для мультиарендности (изоляция данных).
    *   `data` (JSONB): Фактические данные записи.
        *   Содержит значения полей согласно схеме сущности.
    *   `created_by` (UUID): ID пользователя-создателя.
    *   `updated_at`, `created_at`: Таймстампы.
    *   **Индексы:**
        *   `idx_entity_tenant`: `(entity_id, tenant_id)` — для быстрого поиска.
        *   `idx_data_gin`: `USING GIN (data)` — для поиска по содержимому JSON.
        *   `idx_data_field`: Частичные индексы для часто используемых полей (настраиваются через метаданные).

### 3.3. Очередь событий (Outbox Pattern)
Гарантирует доставку событий для воркфлоу.

1.  **`event_outbox`**
    *   `id` (UUID)
    *   `event_type` (String): Например, `record.created`, `record.updated`.
    *   `payload` (JSONB): Данные события.
    *   `status` (Enum): `pending`, `processed`, `failed`.
    *   `created_at`: Время создания.
    *   *Запись происходит в той же транзакции, что и изменение в `records`.*

## 4. Функциональные модули

### 4.1. Модуль управления метаданными (Schema Manager)
*   **API Endpoints:**
    *   `POST /api/v1/entities`: Создать новую сущность.
    *   `PUT /api/v1/entities/{slug}`: Обновить схему (добавить поле, изменить тип).
    *   `GET /api/v1/entities`: Список всех сущностей.
    *   `DELETE /api/v1/entities/{slug}`: Удалить сущность (логически или физически).
*   **Логика:**
    *   Валидация схемы через Pydantic перед сохранением в БД.
    *   Инкремент версии схемы при изменениях.
    *   Проверка конфликтов имен полей.

### 4.2. Динамический CRUD Engine (Query Core)
*   **Принцип:** Перевод HTTP-запросов в SQL-запросы к таблице `records` с использованием JSONB-операторов.
*   **API Endpoints (универсальные):**
    *   `POST /api/data/{entity_slug}`: Создание записи.
        *   Валидация тела запроса против актуальной схемы сущности.
        *   Триггер события `record.created` в outbox.
    *   `GET /api/data/{entity_slug}`: Получение списка.
        *   Поддержка фильтров: `?filter[price][gt]=100&filter[status]=active`.
        *   Трансляция в SQL: `data->>'price' > '100' AND data->>'status' = 'active'`.
        *   Сортировка, пагинация.
        *   Применение RBAC фильтров (автоматическая подстановка условий доступа).
    *   `GET /api/data/{entity_slug}/{id}`: Получение одной записи.
    *   `PATCH /api/data/{entity_slug}/{id}`: Частичное обновление.
        *   Триггер события `record.updated`.
    *   `DELETE /api/data/{entity_slug}/{id}`: Удаление.
*   **Требования:**
    *   Whitelist операторов фильтрации для безопасности.
    *   Лимит глубины вложенных запросов.
    *   Кэширование схемы сущности в Redis (ключ: `schema:{slug}:{version}`).

### 4.3. Workflow Engine (Automation)
*   **Архитектура:** Event-driven. Воркер слушает таблицу `event_outbox`.
*   **Конфигурация:** Хранится в БД (граф шагов: триггер -> условия -> действия).
*   **Действия (Actions):**
    *   Строго ограниченный набор (без выполнения произвольного Python кода):
        *   `http_request`: Отправка вебхука.
        *   `send_email`: Отправка письма (через SMTP/SNS).
        *   `delay`: Пауза выполнения.
        *   `transform_data`: Простые манипуляции с JSON (маппинг полей).
        *   `db_operation`: Создание/обновление другой записи.
*   **Исполнение (ARQ Worker):**
    *   Задачи выполняются асинхронно.
    *   Реализация идемпотентности (повторная обработка не должна ломать данные).
    *   Механизм Retry с экспоненциальной задержкой.

### 4.4. Система безопасности (Auth & RBAC)
*   **Аутентификация:** JWT (Access + Refresh tokens).
*   **Авторизация:**
    *   Middleware проверяет токен и извлекает роль/ID пользователя.
    *   Query Core применяет фильтры доступа к каждому запросу выборки.
    *   Проверка прав на действие (CRUD) перед выполнением операции.
*   **Multi-tenancy:** Все запросы к данным автоматически фильтруются по `tenant_id` из контекста пользователя.

## 5. Инфраструктура и Деплой

### 5.1. Docker Compose
Проект должен запускаться одной командой `docker-compose up -d`.
Состав сервисов:
1.  `db` (PostgreSQL 15)
2.  `redis` (Redis 7)
3.  `api` (FastAPI app)
4.  `worker` (ARQ worker)
5.  `frontend` (Next.js, опционально в compose или отдельно)

### 5.2. Наблюдаемость
*   **Логи:** Вывод в stdout в формате JSON (Structlog). Поля: `timestamp`, `level`, `event`, `request_id`, `user_id`.
*   **Метрики:** Экспорт базовых метрик (количество запросов, latency, ошибки) в формате Prometheus (эндпоинт `/metrics`).
*   **Health Checks:** Эндпоинты `/health` (DB connection) и `/ready`.

## 6. План реализации (Roadmap)

### Этап 1: Фундамент (MVP Core)
*   Настройка Docker Compose (DB, Redis, API skeleton).
*   Реализация моделей Metadata (`entities`) и Data (`records`).
*   Базовый CRUD API для сущностей (без сложной фильтрации).
*   Простая аутентификация (JWT).

### Этап 2: Продвинутые данные и Безопасность
*   Реализация Query Core с поддержкой JSONB фильтров и сортировки.
*   Внедрение RBAC и Row-Level Security.
*   Кэширование схем в Redis.
*   Валидация данных на основе схемы (Pydantic dynamic models).

### Этап 3: Автоматизация (Workflow)
*   Таблица `event_outbox` и механизм транзакционной записи.
*   Настройка ARQ воркера.
*   Реализация базовых действий (Webhook, DB Update).
*   Конфигуратор workflow (хранение графов в БД).

### Этап 4: Production Ready
*   Оптимизация индексов PostgreSQL.
*   Расширенное логирование и метрики.
*   Тесты (Unit + Integration).
*   Документация API (Swagger/Redoc).

## 7. Критерии приемки
1.  Возможность создать сущность "Заказы" с полями (сумма, статус, клиент) через API и сразу начать сохранять туда данные.
2.  Фильтрация заказов по сумме и статусу работает корректно и быстро (используя GIN индекс).
3.  Пользователь с ролью "Менеджер" видит только свои заказы (RBAC filter).
4.  При создании заказа автоматически отправляется вебхук (Workflow).
5.  Система полностью поднимается через `docker-compose up`.
