# Архитектура универсального Low-Code Backend на Python

## 1. Обзор архитектуры

Архитектура построена по модульному принципу с четким разделением ответственности между слоями. Основной подход — **гибридная модель**, сочетающая надежность классического монолита для управления метаданными и асинхронную производительность для обработки пользовательских запросов.

Система состоит из следующих ключевых слоев:
1. **Слой представления (API Gateway & UI)**
2. **Слой оркестрации (Core Engine)**
3. **Слой динамической логики (Runtime Sandbox)**
4. **Слой данных (Dynamic Data Layer)**
5. **Слой инфраструктуры (Infrastructure & Ops)**

---

## 2. Детальное описание компонентов

### 2.1. Слой представления (Presentation Layer)

Этот слой отвечает за взаимодействие с внешним миром: администраторами, конечными пользователями и внешними системами.

#### Компоненты:
*   **Единая точка входа (API Gateway):**
    *   Маршрутизирует входящие запросы либо к системному API (управление конфигуратором), либо к динамическому API (пользовательские сущности).
    *   Выполняет первичную валидацию токенов, лимитирование запросов (Rate Limiting) и базовый парсинг контента.
    *   Поддерживает протоколы REST, GraphQL и WebSocket через единую конфигурацию.
*   **Веб-интерфейс конструктора (Studio UI):**
    *   Single Page Application (SPA) для визуального проектирования моделей данных, настройки прав доступа и создания бизнес-процессов.
    *   Взаимодействует исключительно с системным API.
*   **Генератор клиентских SDK:**
    *   Автоматически генерирует типизированные клиенты для фронтенда на основе актуальной схемы данных.

### 2.2. Слой оркестрации (Core Engine)

«Мозг» системы, управляющий метаданными и координирующий работу всех подсистем. Работает в синхронном или гибридном режиме для гарантии целостности конфигурации.

#### Компоненты:
*   **Реестр метаданных (Metadata Registry):**
    *   Хранит определения сущностей, полей, связей, правил валидации и версионирование схем.
    *   Кэширует активные конфигурации в памяти для ускорения доступа (Hot Reload конфигурации без перезапуска сервиса).
*   **Движок безопасности (Security Core):**
    *   Централизованный сервис проверки прав доступа (RBAC/ABAC).
    *   Перехватывает каждый запрос к данным и оценивает контекст (кто, что, при каких условиях) перед передачей в слой данных.
*   **Планировщик задач (Task Scheduler):**
    *   Управляет отложенными задачами, периодическими джобами (cron) и очередями событий.
    *   Распределяет тяжелые операции по воркерам.

### 2.3. Слой динамической логики (Logic Runtime)

Изолированная среда исполнения пользовательской бизнес-логики. Критически важный компонент для безопасности и стабильности.

#### Компоненты:
*   **Интерпретатор рабочих процессов (Workflow Interpreter):**
    *   Выполняет графы бизнес-процессов, описанные в декларативном формате (JSON/YAML).
    *   Поддерживает узлы: условия, циклы, параллельное выполнение, вызов внешних API.
*   **Песочница кода (Code Sandbox):**
    *   Изолированное окружение для выполнения пользовательских скриптов (Python-сниппетов).
    *   Ограничивает доступ к файловой системе, сети и системным вызовам.
    *   Контролирует потребление ресурсов (CPU, RAM, время выполнения) для предотвращения зависаний.
*   **Система триггеров (Event Bus):**
    *   Шина событий, связывающая изменения данных с запуском бизнес-процессов.
    *   Типы триггеров: `Before Save`, `After Save`, `On Delete`, `On Schedule`, `Webhook`.

### 2.4. Слой данных (Dynamic Data Layer)

Самый сложный слой, обеспечивающий хранение и манипуляцию данными, структура которых неизвестна на момент компиляции приложения.

#### Компоненты:
*   **Динамический маппер (Dynamic ORM Adapter):**
    *   Транслирует абстрактные модели из реестра метаданных в конкретные запросы к СУБД.
    *   Не требует перезагрузки приложения при изменении схемы данных.
    *   Использует возможности СУБД для работы с полуструктурированными данными (JSONB) в сочетании с реляционными связями.
*   **Менеджер миграций (Schema Migration Manager):**
    *   Автоматически применяет изменения структуры таблиц (добавление колонок, индексов) при обновлении модели в конструкторе.
    *   Ведет журнал изменений схемы для возможности отката.
*   **Аудитор данных (Data Auditor):**
    *   Прозрачно логирует все изменения данных (кто, когда, старое значение, новое значение) в отдельное хранилище.

### 2.5. Слой инфраструктуры (Infrastructure)

Базовые сервисы, обеспечивающие работу системы.

#### Компоненты:
*   **Кэш-слой:** Хранение сессионных данных, результатов частых запросов и скомпилированных схем.
*   **Очереди сообщений:** Асинхронная связь между компонентами, буферизация пиковых нагрузок.
*   **Хранилище файлов:** Абстракция над объектным хранилищем для загрузки пользователем файлов (документы, изображения).

## 3. Потоки данных (Data Flow)

### Сценарий А: Создание новой сущности (Конфигурация)
1.  Пользователь в **Studio UI** создает сущность «Заказ» с полями.
2.  Запрос уходит в **API Gateway** -> **Core Engine**.
3.  **Metadata Registry** валидирует схему и сохраняет новую версию метаданных.
4.  **Schema Migration Manager** генерирует и выполняет SQL-скрипт для обновления физической структуры БД.
5.  Кэш схем инвалидируется, новая схема загружается в память.
6.  Пользователь получает подтверждение. Система готова принимать данные в новую сущность.

### Сценарий Б: Обработка пользовательского запроса (Runtime)
1.  Клиент отправляет `POST /api/orders`.
2.  **API Gateway** принимает запрос, извлекает токен.
3.  **Security Core** проверяет права пользователя на создание записи в сущности «Заказ».
4.  Если доступ разрешен, **Dynamic ORM Adapter** строит запрос к БД на основе актуальных метаданных.
5.  Данные сохраняются в БД.
6.  Срабатывает **Event Bus**: фиксируется событие `After Save`.
7.  **Workflow Interpreter** запускает привязанный процесс (например, «Отправить письмо»).
8.  Тяжелая задача отправляется в **Очередь сообщений** для фоновой обработки воркером.
9.  Клиенту возвращается ответ с созданным объектом.

## 4. Принципы обеспечения надежности и безопасности

1.  **Изоляция исполняемого кода:** Пользовательские скрипты выполняются в ограниченном контексте без доступа к внутренностям ядра системы.
2.  **Транзакционность:** Изменение метаданных и применение миграций БД происходят в рамках одной транзакции. При ошибке откатывается всё.
3.  **Версионирование схем:** Любое изменение структуры данных сохраняет предыдущую версию, что позволяет анализировать историю и откатывать изменения.
4.  **Гранулярный аудит:** Полная трассируемость действий как администраторов конструктора, так и конечных пользователей системы.
5.  **Масштабируемость:** Состояние (State) хранится в БД/Redis, сами узлы приложения не хранят состояния (Stateless), что позволяет горизонтально масштабировать слой обработки запросов.

## 5. Технологические требования к реализации

Для реализации данной архитектуры в рамках экосистемы Python требуются компоненты, способные обеспечить:
*   **Динамическую типизацию:** Возможность описывать модели данных «на лету».
*   **Асинхронный ввод-вывод:** Для обработки большого количества одновременных соединений к API.
*   **Безопасное исполнение кода:** Механизмы ограничения ресурсов для песочницы.
*   **Работу с JSONB:** Эффективное использование гибридных моделей данных в реляционной СУБД.
*   **Управление очередями:** Надежная доставка событий для фоновых задач.
