# Техническое задание: Frontend (Low-Code Platform)

## 1. Общие сведения
**Наименование проекта:** Веб-интерфейс управления Low-Code платформой.
**Цель:** Разработка интуитивно понятного Single Page Application (SPA) для визуального конструирования моделей данных, настройки бизнес-логики, управления доступом и просмотра данных.
**Тип приложения:** Dashboard / Admin Panel с элементами конструктора (Drag-n-Drop).

## 2. Стек технологий
*   **Фреймворк:** React 18+ / Vue 3 / SvelteKit (на выбор команды, рекомендация: React + TypeScript).
*   **Язык:** TypeScript (строгая типизация обязательна).
*   **State Management:** Zustand / Redux Toolkit / Pinia.
*   **UI Kit:** Ant Design / MUI / Tailwind UI (готовые компоненты для ускорения разработки).
*   **Формы и валидация:** React Hook Form + Zod / Yup.
*   **HTTP Client:** Axios / TanStack Query (React Query) для кэширования запросов.
*   **Сборка:** Vite.
*   **Контейнеризация:** Nginx (Alpine) для раздачи статики в Docker Compose.

## 3. Архитектура приложения
Приложение строится по модульному принципу:
1.  **Core:** Настройка роутинга, провайдеров (Auth, API), глобальных тем.
2.  **Modules:** Изолированные доменные области (Data Models, Logic Builder, Users, Data Viewer).
3.  **Shared UI:** Переиспользуемые компоненты (кнопки, инпуты, модалки).
4.  **Services:** Слой абстракции над API бэкенда.

## 4. Функциональные требования по модулям

### 4.1. Модуль авторизации и навигации
*   **Страницы:** Login, Forgot Password.
*   **Layout:** Боковое меню (Sidebar), верхняя панель (Header) с профилем пользователя.
*   **Роутинг:** Защита роутов на основе ролей (Admin, Developer, Viewer). Динамическое меню в зависимости от прав.
*   **Сессия:** Автоматическое обновление JWT токенов, редирект на логин при истечении сессии.

### 4.2. Конструктор моделей данных (Data Model Builder)
*   **Список моделей:** Таблица всех созданных сущностей с поиском и фильтрацией.
*   **Редактор модели:**
    *   Визуальная форма добавления/редактирования полей (Имя, Тип, Обязательность, Дефолтное значение).
    *   Поддержка типов: String, Integer, Float, Boolean, DateTime, JSON, Relation (One-to-One, One-to-Many, Many-to-Many).
    *   Валидация имен полей на лету (regex, уникальность).
*   **Управление связями:** Визуализация связей между моделями (опционально граф или список).
*   **Превью схемы:** Отображение JSON-представления модели перед сохранением.

### 4.3. Конструктор бизнес-логики (Logic Flow Builder)
*   **Список триггеров:** Просмотр существующих хуков и скриптов.
*   **Визуальный редактор (No-Code):**
    *   Drag-n-Drop интерфейс для построения цепочек действий.
    *   Блоки: "Условие" (If/Else), "Действие" (Create/Update Record), "Отправка Email", "HTTP Запрос".
    *   Соединение блоков линиями (Flow-based).
*   **Редактор кода (Low-Code):** Встроенный редактор (Monaco Editor) для написания Python-скриптов с подсветкой синтаксиса и автодополнением (если бэкенд предоставляет метаданные).
*   **Тестирование:** Кнопка "Запустить тест" с моковыми данными.

### 4.4. Управление доступом (Access Control)
*   **Менеджер ролей:** Создание/удаление ролей.
*   **Матрица прав:** Таблица (Роли x Модели), где чекбоксами выставляются права (C, R, U, D).
*   **Полевая безопасность:** Детальная настройка видимости/редактируемости конкретных полей для выбранной роли.

### 4.5. Просмотрщик данных (Data Browser)
*   **Динамические таблицы:** Автоматическая генерация таблицы данных на основе выбранной модели.
*   **Фильтрация и сортировка:** UI элементы для фильтрации колонок, сортировки по клику на заголовок.
*   **CRUD операции:**
    *   Кнопка "Создать запись" (открывает модальное окно с динамической формой).
    *   Кнопки "Редактировать" и "Удалить" в строке таблицы.
*   **Пагинация:** Постраничная навигация с выбором размера страницы.
*   **Экспорт:** Кнопка выгрузки данных в CSV/JSON.

### 4.6. Системный дашборд и логи
*   **Мониторинг:** Графики количества запросов, ошибок, активных пользователей.
*   **Логи:** Просмотр журнала аудита (кто что изменил) и логов выполнения скриптов с фильтрацией по уровням (Error, Warning, Info).

## 5. Нефункциональные требования
*   **UX/UI:** Адаптивный дизайн (Desktop first, но с поддержкой планшетов). Темная/Светлая тема.
*   **Производительность:** First Contentful Paint < 1.5с. Отклик интерфейса на действия пользователя < 100мс.
*   **Масштабируемость:** Код должен быть готов к росту количества моделей (рендеринг больших списков через виртуализацию).
*   **Безопасность:** XSS защита (санитизация ввода), хранение токенов в httpOnly cookies или secure storage.
*   **Документированность:** Storybook для компонентов (опционально, но желательно).

```text
├── Dockerfile

├── tsconfig.json

```

## 7. Этапы разработки (Roadmap)
1.  **MVP (Спринт 1):**
    *   Настройка проекта, линтеры, базовый лейаут.
    *   Страница логина и интеграция с Auth API.
    *   Список моделей и простой CRUD создания модели (без связей).
2.  **Спринт 2:**
    *   Полноценный редактор моделей (все типы полей, валидация).
    *   Data Browser (просмотр данных созданных моделей).
3.  **Спринт 3:**
    *   Модуль управления ролями и правами доступа (матрица прав).
    *   Интеграция с системными логами.
4.  **Спринт 4:**
    *   Визуальный конструктор логики (базовая версия).
    *   Полировка UI, адаптивность, темизация.
    *   Оптимизация производительности.

## 8. Критерии приемки
*   Приложение собирается в Docker-образ и стартует через `docker-compose`.
*   Пользователь может создать модель с полями разных типов через UI.
*   Созданная модель сразу отображается в разделе "Данные" с возможностью CRUD операций.
*   Настройки прав доступа реально ограничивают видимость пунктов меню и кнопок в UI.
*   Интерфейс корректно обрабатывает ошибки сети и валидации бэкенда.
*   Код покрыт типами TypeScript (минимум 90% файлов).
