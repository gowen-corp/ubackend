# Техническое задание: Frontend (Studio UI)

## 1. Обзор проекта
**Цель:** Разработка веб-интерфейса (Studio UI) для визуального управления low-code платформой. Интерфейс позволяет пользователям создавать сущности, настраивать поля, проектировать бизнес-процессы и управлять данными без написания кода.

**Ключевая особенность:** Динамическая генерация форм и интерфейсов на основе метаданных, полученных от backend.

## 2. Технологический стек
| Компонент | Технология | Обоснование |
| :--- | :--- | :--- |
| **Framework** | Next.js 14+ (App Router) | SSR/SSG, отличная производительность, роутинг из коробки |
| **Язык** | TypeScript | Строгая типизация, безопасность кода, автодокументирование |
| **UI Library** | shadcn/ui + Radix UI | Доступные компоненты, полная кастомизация, не оверхед как MUI |
| **Стилизация** | Tailwind CSS | Ускорение верстки, консистентность дизайна |
| **State Management** | Zustand | Простой и легковесный, идеален для сложного состояния конструктора |
| **Формы** | React Hook Form + Zod | Эффективная работа с динамическими формами, валидация на клиенте |
| **HTTP Client** | Axios / TanStack Query | Кэширование запросов, оптимистичные обновления, retry-логика |
| **Контейнеризация** | Docker (в составе docker-compose) | Единая среда запуска с backend |

## 3. Архитектура приложения

### 3.1. Структура проекта
```
frontend/
├── app/                    # Next.js App Router
│   ├── (auth)/             # Группа маршрутов аутентификации
│   │   ├── login/
│   │   └── register/
│   ├── (dashboard)/        # Основная часть после логина
│   │   ├── entities/       # Конструктор сущностей
│   │   ├── data/           # Просмотр/редактирование данных
│   │   ├── workflows/      # Редактор бизнес-процессов
│   │   ├── users/          # Управление пользователями и ролями
│   │   └── settings/       # Настройки проекта
│   └── layout.tsx
├── components/
│   ├── ui/                 # Базовые компоненты (shadcn)
│   ├── entities/           # Компоненты конструктора сущностей
│   ├── forms/              # Динамические формы
│   ├── workflows/          # Визуализатор графов workflow
│   └── data-table/         # Универсальная таблица данных
├── lib/
│   ├── api.ts              # API client
│   ├── utils.ts            # Утилиты
│   └── validators.ts       # Zod схемы
├── stores/
│   ├── entityStore.ts      # Zustand store для сущностей
│   └── authStore.ts        # Zustand store для аутентификации
└── types/
    └── index.ts            # TypeScript типы
```

### 3.2. Управление состоянием
*   **Глобальное состояние:** Zustand stores для хранения метаданных сущностей, текущего пользователя, настроек UI.
*   **Серверное состояние:** TanStack Query для кэширования API-запросов, фоновой синхронизации, обработки loading/error состояний.
*   **Локальное состояние:** React useState/useReducer для компонентов.

## 4. Функциональные модули

### 4.1. Модуль аутентификации
*   **Страницы:**
    *   Login (email/password)
    *   Register (для первого входа админа)
    *   Forgot Password
*   **Функционал:**
    *   Сохранение JWT токена (httpOnly cookie или localStorage).
    *   Автоматический refresh токена.
    *   Protected routes (редирект на login если нет токена).
    *   Отображение профиля пользователя в хедере.

### 4.2. Конструктор сущностей (Entity Builder)
**Ключевой модуль платформы.**

*   **Список сущностей:**
    *   Таблица всех созданных сущностей (slug, name, количество записей, дата создания).
    *   Кнопки: Создать, Редактировать, Удалить.
*   **Редактор сущности:**
    *   **Основные поля:** Название (name), Slug (автоматическая генерация из name).
    *   **Конструктор полей (Field Builder):**
        *   Добавление/удаление/перемещение полей.
        *   Типы полей: String, Number, Boolean, Date, JSON, Relation (ссылка на другую сущность).
        *   Настройки каждого поля:
            *   Label (отображаемое имя)
            *   Key (техническое имя, unique)
            *   Required (обязательное)
            *   Default value
            *   Validation rules (min, max, pattern, enum values)
    *   **Предпросмотр схемы:** JSON view текущей схемы.
    *   **Сохранение:** Отправка POST/PUT запроса на `/api/v1/entities`.
*   **Валидация:**
    *   Проверка уникальности slug и field keys.
    *   Запрет на удаление сущности с записями (или подтверждение каскадного удаления).

### 4.3. Управление данными (Data Browser)
**Динамический интерфейс для CRUD операций над записями пользователей.**

*   **Автоматическая генерация UI:**
    *   На основе `schema_definition` сущности генерируется:
        *   Таблица со столбцами (sortable, filterable).
        *   Форма создания/редактирования записи.
        *   Фильтры для каждого поля (текстовый поиск, диапазон чисел, выбор даты, dropdown для enum).
*   **Функционал таблицы:**
    *   Пагинация (server-side).
    *   Сортировка по любому полю (кликом на заголовок).
    *   Массовые действия (выбрать несколько, удалить).
    *   Быстрое редактирование ячеек (inline edit).
*   **Форма записи:**
    *   Динамическая генерация инпутов в зависимости от типа поля:
        *   String → Input / Textarea
        *   Number → Number Input
        *   Boolean → Switch / Checkbox
        *   Date → Date Picker
        *   Enum → Select / Radio Group
        *   Relation → Async Select (поиск по связанной сущности)
    *   Валидация в реальном времени (Zod схема, сгенерированная из metadata).

### 4.4. Редактор Workflow (Automation Builder)
**Визуальный конструктор бизнес-процессов.**

*   **Интерфейс:**
    *   Canvas для перетаскивания блоков (drag-and-drop).
    *   Панель инструментов слева (триггеры, действия, условия).
    *   Панель свойств справа (настройка выбранного блока).
*   **Элементы:**
    *   **Триггеры:** `Record Created`, `Record Updated`, `Scheduled`, `Webhook Received`.
    *   **Действия:** `HTTP Request`, `Send Email`, `Update Record`, `Delay`, `Transform Data`.
    *   **Логика:** `Condition` (if/else), `Loop` (опционально для MVP).
*   **Визуализация:**
    *   Связи между блоками (линии).
    *   Статус выполнения (для отладки).
*   **Сохранение:** Граф сохраняется как JSON в backend.

### 4.5. Управление доступом (RBAC Manager)
*   **Список ролей:** Admin, Manager, User (кастомные роли).
*   **Редактор роли:**
    *   Выбор сущности.
    *   Чекбоксы действий: Create, Read, Update, Delete.
    *   Поле для Row-Level фильтра (JSON input с подсказками).
*   **Назначение ролей пользователям:**
    *   Таблица пользователей с dropdown выбора роли.

### 4.6. Настройки и Профиль
*   Профиль пользователя (смена пароля, email).
*   Настройки проекта (название, логотип).
*   Audit Log (просмотр логов действий пользователей).

## 5. Интеграция с Backend

### 5.1. API Client
*   Базовый URL берется из переменных окружения (`NEXT_PUBLIC_API_URL`).
*   Автоматическое добавление JWT токена в заголовок `Authorization`.
*   Обработка ошибок:
    *   401 → редирект на login.
    *   403 → показ toast "Нет прав".
    *   500 → показ toast "Ошибка сервера".

### 5.2. Работа с динамическими схемами
*   При загрузке страницы сущности делается запрос `GET /api/v1/entities/{slug}`.
*   Полученная `schema_definition` используется для:
    *   Генерации Zod схемы валидации.
    *   Рендеринга соответствующих компонентов форм.
    *   Построения столбцов таблицы.

### 5.3. Real-time обновления (опционально)
*   Использование WebSocket или polling для обновления списка записей при изменениях другими пользователями (для MVP можно пропустить).

## 6. Требования к UI/UX

### 6.1. Дизайн-система
*   Использование компонентов shadcn/ui как базы.
*   Цветовая схема: светлая/темная тема (переключатель).
*   Адаптивность: корректное отображение на планшетах (desktop-first).

### 6.2. Обратная связь
*   **Loading states:** Скелетоны для таблиц и форм во время загрузки.
*   **Toast уведомления:** Успешное сохранение, ошибки валидации, ошибки сети.
*   **Подтверждения:** Dialog перед удалением сущности или записи.

### 6.3. Производительность
*   Lazy loading для тяжелых компонентов (редактор workflow).
*   Виртуализация списков при большом количестве записей (tanstack virtual).
*   Оптимистичные обновления UI при CRUD операциях.

## 7. План реализации (Roadmap)

### Этап 1: Каркас и Аутентификация
*   Инициализация Next.js проекта с TypeScript и Tailwind.
*   Установка shadcn/ui, настройка базовых компонентов.
*   Реализация страниц Login/Register.
*   Настройка API client и защищенных роутов.

### Этап 2: Конструктор сущностей
*   Создание страницы списка сущностей.
*   Разработка редактора сущностей (добавление полей, настройка типов).
*   Интеграция с backend API (`/api/v1/entities`).
*   Валидация схем на клиенте.

### Этап 3: Управление данными (Dynamic Forms)
*   Создание универсального компонента DataTable.
*   Разработка генератора форм на основе schema_definition.
*   Реализация фильтров и сортировки.
*   Интеграция CRUD операций для записей.

### Этап 4: Workflow и RBAC
*   Простой визуальный редактор workflow (можно использовать react-flow).
*   Интерфейс управления ролями и правами.
*   Финальное тестирование и полировка UI.

## 8. Критерии приемки
1.  Пользователь может войти в систему и увидеть дашборд.
2.  Можно создать сущность "Продукты" с полями (название, цена, категория) через UI.
3.  После создания сущности автоматически появляется страница для управления записями этой сущности.
4.  Можно создать, отредактировать, удалить запись в сущности "Продукты".
5.  Фильтрация и сортировка записей работают корректно.
6.  Можно настроить простой workflow: при создании продукта отправлять webhook.
7.  Интерфейс адаптирован под разные разрешения экрана.
8.  Проект запускается через `docker-compose up` вместе с backend.
